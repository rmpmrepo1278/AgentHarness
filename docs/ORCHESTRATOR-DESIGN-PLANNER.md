## Problem Statement

Hermes's LLM proxy routes requests using a flat priority list based on word count and tool presence. It has no concept of cognitive task types (reasoning vs execution vs chat). Every request, whether "fix n8n, it's in a boot loop" or "what time is it," gets the same treatment: try Groq first, then cascade through providers until one works.

This means:
- Simple chat burns expensive model capacity when free/local would suffice
- Complex multi-step problems get sent to cheap models that can't reason about them
- There's no ability to decompose a problem into a plan and execute it step by step
- Infrastructure failures require manual intervention, even when the fix is routine

The goal: build a planner-executor orchestrator inside Hermes that classifies requests by cognitive tier, decomposes complex tasks into structured plans, executes each step with the cheapest capable model, and eventually acts proactively on infrastructure events before the user even notices.

## What Makes This Cool

Your homelab fixes itself. You send "fix n8n" via Telegram and watch Hermes autonomously investigate, plan, execute, and report back. And it costs almost nothing because only the planning steps hit expensive models. The execution (running commands, reading logs, restarting containers) all goes through free/local models.

The 10x version: you don't even send the message. HomeButler detects n8n crashed at 2am, the orchestrator plans and executes the fix, and by morning you get a Telegram summary: "n8n had a DB connection issue. I restarted postgres and verified n8n is healthy. Here's what happened."

## Constraints

- **Hardware:** Existing homelab server (HP box, 192.168.29.10). Local LLM running at :8081 (currently Qwen 2.5 7B, CPU-only inference, 4096 ctx).
- **Budget:** Under $10/month for paid API calls. Free-first routing is non-negotiable.
- **Single agent:** Hermes is the sole agent. No multi-agent swarm. The orchestrator lives inside Hermes.
- **Existing infra:** Proxy at :8080, HomeButler Docker event monitoring, Telegram gateway, all existing tools (terminal, browser, file ops, memory, skills).
- **Reliability:** Hermes runs 24/7 via systemd. Orchestrator must be fault-tolerant (graceful fallback if a planning step fails).
- **Nightly network outage:** T-Mobile gateway powers off nightly (~1am-2am+). All cloud API calls fail during this window. The orchestrator must be network-aware: queue predictive triggers during outage, process them when connectivity resumes. Distinguish "quiet hours" (policy: suppress autonomous writes) from "network outage" (hard constraint: no API access).

## Premises

1. **Local Qwen 2.5 7B can classify requests reliably.** Classification (chat vs execute vs reason) is a simple task. Small models handle it well with a good system prompt. **GATE: Phase 1 is a two-step delivery.** Build the classifier, run accuracy test on 50 messages. Only proceed with proxy integration if accuracy > 90%. Fallback: simplify to 2-way (SIMPLE vs COMPLEX) or use Groq for classification.
2. **Strong reasoning models produce consistent JSON plans.** Gemini 2.5 Pro, Kimi K2.6, and similar models can output structured JSON plans when given a strict schema. Mitigation: JSON schema validation + retry on parse failure.
3. **The proxy can route by cognitive tier with a hint field.** Adding a `cognitive_tier` parameter to the proxy request that overrides the default routing cascade. Minimal proxy change.
4. **Predictive orchestration builds on existing HomeButler events.** The Docker event stream, container health checks, and event history are already being collected. The gap is: nobody is watching and acting on patterns.
5. **This ships incrementally.** Phase 1 (classifier) works standalone. Phase 2 (plan-execute loop) builds on it. Phase 3 (predictive) adds event-driven triggers. Each phase adds independent value.

## Approaches Considered

### Approach A: Smart Classifier Only
Add a local LLM classifier to Hermes that tags each turn as chat/execute/reason. Proxy routes to different model tiers. No plan-execute loop.
- Effort: S (~150 lines, 1-2 files)
- Risk: Low
- Ships fast but doesn't deliver the "fix n8n" demo

### Approach B: Planner-Executor Loop
Full implementation: classifier + reasoning model creates JSON plan + execution model runs steps + feedback loop. All inside Hermes run_agent.py.
- Effort: M (~400 lines, 3-4 files)
- Risk: Medium (JSON parsing fragility)
- The full demo works end-to-end

### Approach C: Predictive Planner-Executor (CHOSEN)
Approach B + event-driven triggers from HomeButler. Proactive detection, planning, execution, and reporting.
- Effort: L (~600 lines, 5-6 files, phased)
- Risk: Medium-High (autonomous actions need safety rails)
- The "whoa" demo: homelab fixes itself

## Recommended Approach

**Approach C, built in three phases.** Each phase ships independently and adds value.

### Phase 1: Cognitive Classifier + Tier Routing

**Where:** `~/.hermes/hermes-agent/agent/cognitive_router.py` (new file) + `proxy_server.py` (minor change)

**What it does:**
- Before each model call, Hermes sends the user's message to the local LLM with a classification prompt
- Local LLM returns one of: `CHAT`, `EXECUTE`, `REASON`, `PLAN_NEEDED`
- Hermes passes this as a `cognitive_tier` field in the JSON request body (alongside the standard OpenAI fields: `model`, `messages`, `tools`, etc.). The proxy reads `body.get("cognitive_tier")` and uses it to reorder the provider cascade. Non-standard field, ignored by upstream providers.
- Proxy routes: CHAT → local/free, EXECUTE → tool-calling cascade (Groq > Google-alt > Cerebras, current order), REASON → Gemini Pro first, PLAN_NEEDED → triggers Phase 2
- **Both code paths** (`_tool_call_passthrough` for tool requests AND the plain-chat router path) must read the tier hint

**Classification prompt (for local LLM):**
```
Classify this user request into exactly one category:
- CHAT: casual conversation, greetings, questions about yourself
  Examples: "hello", "how are you", "what can you do", "thanks"
- EXECUTE: a single action that can be done with one tool call
  Examples: "restart n8n", "show me docker ps", "read /etc/hosts", "check disk space"
- REASON: needs analysis or explanation but can be done in one model response
  Examples: "why is my DNS slow", "explain the proxy routing", "what does this error mean"
- PLAN_NEEDED: complex task requiring multiple steps, investigation, or fixing something broken
  Examples: "fix n8n it's in a boot loop", "set up a new service", "debug why backups are failing", "migrate the database"

When uncertain, default to the cheaper tier (CHAT > EXECUTE > REASON > PLAN_NEEDED).

User request: "{message}"

Category:
```

**Proxy change:** Both `_tool_call_passthrough` and the plain-chat routing path check for `body.get("cognitive_tier")`. If present, reorder `_TOOL_PROVIDERS` to prioritize the right tier:
- CHAT: local LLM only, strip tools
- EXECUTE: tool-calling cascade (Groq > Google-alt > Cerebras > SambaNova > OpenRouter > Google-primary). Note: SambaNova has a known empty-response bug after tool calls. If a plan step fails on SambaNova, the orchestrator retries with the next provider, not the same one.
- REASON: Gemini Pro first
- PLAN_NEEDED: Gemini Pro first (plan creation)

**Estimated cost impact:** Most Hermes messages are CHAT or EXECUTE. With classification, ~70% of traffic stays on free/local models. Only REASON and PLAN_NEEDED hit paid Gemini. Projected savings: 60-70% reduction in paid API calls.

### Phase 2: Plan-Execute Loop

**Where:** `~/.hermes/hermes-agent/agent/orchestrator.py` (new file)

**What it does:**
When the classifier returns `PLAN_NEEDED`:

1. **Plan creation:** Send the full conversation context to the reasoning model with a planning prompt. The model returns a JSON plan:
```json
{
  "goal": "Fix n8n boot loop",
  "steps": [
    {"id": 1, "action": "Check n8n container logs for error messages", "tool": "terminal", "command": "docker logs n8n --tail 50"},
    {"id": 2, "action": "Check if postgres is running", "tool": "terminal", "command": "docker ps | grep postgres"},
    {"id": 3, "action": "Analyze findings and determine root cause", "requires_reasoning": true},
    {"id": 4, "action": "Execute fix based on root cause", "depends_on": 3}
  ],
  "success_criteria": "n8n container is running and responding to health checks"
}
```

2. **Step execution:** Loop through steps:
   - Steps with `tool` and `command`: route to cheap/free execution model via proxy
   - Steps with `requires_reasoning`: route back to reasoning model with accumulated results
   - Steps with `depends_on`: wait for dependency to complete

3. **Selective feedback loop:** NOT every step triggers a reasoning-model call. Only these steps get feedback evaluation:
   - Steps with `requires_reasoning: true`
   - Steps that produce errors or unexpected output (non-zero exit code, empty output when output expected)
   - The final step (to confirm success)
   
   For each feedback call, send only: the original goal, the current step, and the last 2 step outputs (not full history). This caps feedback token consumption.
   
   The reasoning model can respond with:
   - `CONTINUE` (proceed to next step)
   - `MODIFY` + updated remaining steps (plan mutation, only add/modify/remove future steps)
   - `ABORT` + reason (stop execution, report failure)
   - `DONE` + summary (declare success early)

4. **Completion:** When all steps complete (or the reasoning model says DONE/ABORT), compose a summary and report to user via Telegram.

**Plan state persistence:** Active plans are persisted to `~/.hermes/orchestrator/active_plan.json`. Each step completion updates the file. On Hermes restart, check for incomplete plans: if found, report the interrupted state to user via Telegram and offer to resume or discard.

**Plan schema validation:** Use Python's `json.loads()` + a simple schema check function. On parse failure, retry with the reasoning model once. On second failure, fall back to non-orchestrated execution (current behavior).

**Safety rails:**
- Maximum 10 steps per plan (prevents runaway loops)
- Maximum 3 feedback iterations per step
- **Destructive commands** require Telegram confirmation. Blocklist (pattern-based):
  - Data loss risk: `rm -rf`, `docker system prune`, `DROP`, `TRUNCATE`, any write to compose files
  - Service disruption risk: `docker stop`, `docker compose down`, `systemctl stop`, `kill -9`
  - Config mutation risk: `sed -i`, `mv`, `cp` to system paths, `chmod`, `chown`
  - Pattern: any command matching `/rm\s+(-[rRf]+|--force|--recursive)/` or `/docker\s+(stop|rm|system)/`
- Plan execution timeout: 10 minutes total (configurable per plan, reasoning model can set `estimated_duration_s` in plan JSON)
- Per-step timeout: 60 seconds for execution steps, 30 seconds for reasoning feedback calls
- **Network-aware:** Before starting a plan that requires API calls, ping the proxy health endpoint. If unreachable, queue the plan for later execution.

**Token economics estimate:** A typical 5-step plan costs approximately:
- Plan creation: ~3K input tokens (context + schema) + ~500 output = ~$0.005
- 2 feedback calls (only error/reasoning steps): ~2K input each + ~200 output = ~$0.004
- Total per plan: ~$0.01. At 5-10 plans/day = $0.05-0.10/day = $1.50-3.00/month. Well within budget.

**Key design decision: The proxy stays dumb.** The orchestrator is entirely inside Hermes. The proxy just sees individual requests with tier hints. This means:
- The proxy doesn't need to understand plans or loops
- Testing is easier (test the orchestrator in isolation)
- The proxy can serve other clients without orchestrator awareness

### Phase 3: Predictive Orchestration

**Where:** `~/.hermes/hermes-agent/agent/predictive_monitor.py` (new file) + HomeButler plugin enhancement

**What it does:**

1. **Event watcher:** A background thread in Hermes monitors HomeButler's event log (`/home/rohit/shared_agent_memory/homebutler.log`). Watches for patterns:
   - Container crash (status: `die`, `oom`)
   - Repeated restarts (>3 in 10 minutes)
   - Health check failures
   - Resource exhaustion signals

2. **Proactive trigger:** When a pattern matches, the monitor creates a synthetic "user request" and feeds it to the orchestrator:
   ```
   [PREDICTIVE] Container 'n8n' has crashed 3 times in the last 10 minutes.
   Last crash reason: OOM. Investigate and fix if possible.
   ```

3. **Autonomous execution with guardrails:**
   - Destructive actions always require Telegram confirmation
   - Read-only investigation (logs, status checks) runs autonomously
   - The plan is sent to Telegram for review before execution of any write actions
   - If no response in 5 minutes, log the plan and wait (don't auto-execute writes during quiet hours)

4. **Morning summary:** A cron job (already have daily digest at 8am) compiles all predictive actions taken overnight into a Telegram summary:
   ```
   Good morning! Here's what happened overnight:
   - 2:14am: n8n crashed (OOM). I restarted it and increased memory limit in compose.
   - 4:30am: Pi-hole DNS latency spike. Investigated, found cache was full. Restarted.
   - All services healthy as of 8:00am.
   ```

**Network outage and quiet hours:**
- **Network outage window** (hard constraint): When cloud APIs are unreachable, queue all predictive triggers to `~/.hermes/orchestrator/pending_triggers.json`. Process the queue when connectivity resumes (check every 60s).
- **Quiet hours** (policy): During quiet hours, read-only investigation runs autonomously. Write actions are queued for morning review. These are separate concepts that may overlap.

**Prerequisites (from homelab upgrade plan):** Phase 3 requires: (a) working backups for all services the orchestrator might touch (so autonomous actions can be reverted), (b) a simple rollback mechanism ("revert last orchestrator action" Telegram command). Phase 3 should not ship until these are in place.

**Morning summary is Phase 3b.** Phase 3a ships the predictive trigger + autonomous investigation. The morning summary (compiling overnight actions into a digest) ships as a follow-up once predictive actions are reliably producing structured output.

## Open Questions

1. **Which reasoning model for planning?** Gemini 2.5 Pro (current, reliable) vs Kimi K2.6 (newer, strong reasoning) vs DeepSeek R1 (open-weight, could run locally eventually). Start with Gemini 2.5 Pro since it's already integrated.

2. **Plan format:** Should plans use a fixed JSON schema or allow the reasoning model more flexibility? Fixed schema is easier to parse but limits the model's ability to express complex logic.

3. **Classification accuracy:** How good is Qwen 2.5 7B at the 4-way classification? Need to test with 50+ example messages and measure accuracy before shipping.

4. **Predictive false positives:** How do you handle the orchestrator "fixing" something that wasn't actually broken? Need a rollback mechanism or at least a "revert last action" command.

5. **Token economics:** How many tokens does a typical planning prompt consume? The conversation context + system prompt + plan schema could be large. Need to measure and potentially compress context for planning calls.

## Success Criteria

- **Phase 1:** Classification accuracy > 90% on a test set of 50 messages. Chat messages never hit paid models. Measured cost reduction > 50%.
- **Phase 2:** The "fix n8n" demo works end-to-end. Plan created, steps executed, results reported. Under 60 seconds for a 5-step plan. Fallback to non-orchestrated mode on any failure.
- **Phase 3:** At least 3 real infrastructure events detected and handled proactively in the first week. Morning summary delivered. No destructive actions without confirmation.
