The Big Picture

What AgentHarness does and why it exists

Your homelab has a brain

Imagine you have an HP laptop sitting on a shelf, running 24/7. It hosts a document manager, a media library, an RSS reader, backups, and several AI agents that talk to you on Telegram. That is Rohit's homelab.

AgentHarness is the invisible middle layer that keeps all of it running. Think of it like a building superintendent: it checks the heating, fixes leaks, routes the mail, and calls for help when something is truly broken.

Why this matters

Most homelabs are fragile. One crashed container at 3 AM means you wake up to broken services. AgentHarness makes the system self-healing -- it detects problems and fixes them before you notice.

The key constraint: resilience on a budget

This is not a cloud server with five nines of uptime. It is a laptop on a shelf with a nightly internet outage. AgentHarness was designed around three hard realities:

🌙

Nightly Blackout

Internet goes down 11 PM - 7:15 AM. Alerts are queued, downloads are deferred, offline-safe tasks run instead.

💰

Free Tier First

Cloud LLM providers have rate limits and quotas. The proxy juggles 7 providers to maximize free usage before spending a cent.

🔨

Self-Healing Required

Nobody wants to SSH in at 3 AM. The system detects failures and fixes them automatically, or at least takes a snapshot before trying.

🚀

Single Machine

Everything runs on one laptop -- 20+ Docker containers, a local LLM, agents, monitoring. Resources must be shared carefully.

The LLM Proxy

Smart routing across 7 AI providers on a budget

One address, seven kitchens

Imagine you are ordering food. Instead of choosing a restaurant, you call one number. Behind that number, a dispatcher knows which kitchens are open, which are busy, which are free today, and which have the best cook for your dish. You just say what you want; the dispatcher figures out where to send it.

That is exactly what proxy_server.py does. Every LLM request on the homelab goes to http://localhost:8080, and the proxy decides which provider actually handles it.

Why not just use one provider?

Free tiers have limits. Groq gives ~200 requests/day. Google rate-limits aggressively. By spreading requests across 7 providers, the homelab gets thousands of free AI calls per day instead of hundreds.

Complexity-based routing

Not all requests are equal. A quick "summarize this" needs a fast, cheap model. A complex coding task needs the best model available. The proxy assigns a complexity level and routes accordingly.

Complexity Provider Priority (first to last) When Used
LOW Local Gemma 4 → Google → Cerebras Short prompts under 20 words
MEDIUM Local → Google → Cerebras → OpenRouter Prompts with 20-100 words
HIGH Groq → Google → Ollama Cloud → SambaNova → Local Prompts over 100 words
CRITICAL Groq → Google → Ollama Cloud → Cerebras → SambaNova → OpenRouter → Local Agent planning, complex reasoning

Inside the router

Here is the actual code that picks a provider. It tries each candidate in order and falls through to the next if one fails.

CODE 
def route(self, request):
    complexity_key = request.complexity.value
    candidate_names = self._routing.get(
        complexity_key, [])

    for name in candidate_names:
        provider = self._providers_by_name.get(name)
        if provider is None:
            continue
        resp = self._try_provider(provider, request)
        if resp is not None:
            return resp

    return LLMResponse.error(
        "router", "No provider available")
PLAIN ENGLISH

Look up what complexity level this request is (LOW, MEDIUM, HIGH, or CRITICAL).

Get the ordered list of providers for that complexity level.

 

 

Go through each provider in order...

Find the provider object by name.

If this provider is not configured, skip it.

 

Try sending the request to this provider.

If the provider returned a good response, we are done -- send it back.

 

 

If we exhausted all providers, return an error saying nobody could handle it.

Rate limits and cooldowns

When a provider returns a 429 (rate limited), the proxy puts it in a 60-second cooldown. If a provider returns 402 (no credits), it is permanently disabled for the session.

CODE 
if resp.status_code == 429:
    _rate_cooldowns[pname] = (
        time.monotonic() + _COOLDOWN_SECONDS)
    continue
if resp.status_code == 402:
    _disabled_providers.add(pname)
    continue
PLAIN ENGLISH

If the provider says "too many requests" (429)...

...mark it as unavailable for 60 seconds (a cooldown timer).

 

Then try the next provider.

If the provider says "no credits left" (402)...

...permanently disable it for this entire session.

Then try the next provider.

Tool passthrough mode

When agents need function calling (like browsing the web or reading files), the proxy switches to a special passthrough mode. It forwards the full tool definitions to cloud providers that support them, capping at 6 tools max to avoid overwhelming free-tier models.

The Doctor

Self-healing infrastructure that fixes itself at 3 AM

An immune system for your server

Your body does not wait for you to notice a cold. White blood cells detect invaders and fight them automatically. Only when something is truly serious does it escalate to pain (so you go to a doctor). The AgentHarness Doctor works the same way.

It has three levels of response: checks that detect problems, runbooks that execute pre-written fixes, and a troubleshooter that gives you step-by-step instructions when automatic fixes are not enough.

Inside a runbook

Runbooks are written in YAML and live in the core/doctor/runbooks/ directory. Each one describes what to check, what to fix, and who to notify.

CODE 
def execute(self, runbook_name, trigger_context):
    cooldown_result = self._check_cooldown(
        runbook_name, start, trigger_context)
    if cooldown_result is not None:
        return cooldown_result

    if not self._acquire_lock(lock_name):
        return # lock held by another runbook

    for step in steps:
        sr = self.execute_step(step, context)
        if not sr.success:
            if "on_fail" in step:
                self._handle_on_fail(step, context)
PLAIN ENGLISH

When asked to run a runbook...

First check: have we tried this fix too many times recently? (Max 3 attempts per 10 minutes.)

 

If we are in cooldown, stop here. Don't hammer the same fix.

 

 

Get an exclusive lock so two runbooks don't try to fix the same service at once.

 

 

Walk through each step in the runbook one by one...

Execute the step (could be a shell command, a health check, or an LLM call).

If the step failed...

...check if the runbook has an alternate path for failures (branching logic).

If so, follow the failure branch instead.

The troubleshooter: when auto-fix is not enough

Some problems need a human. The troubleshooter runs 13 deterministic checks (no AI involved) and generates step-by-step fix instructions for anything it finds.

1

Detect

Runs checks for state files, directory permissions, Docker, Python version, stale locks, circuit breakers, heartbeat freshness, integrity, and LLM provider availability.

2

Diagnose

Each issue gets a root cause analysis. For example: "Directory owned by root, AgentHarness runs as user" or "Scheduler process crashed without cleaning up its locks."

3

Prescribe

Outputs exact shell commands to fix each issue, with a verify command so you can confirm it worked. Copy, paste, done.

Scheduler & MCP Gateway

Network-aware scheduling and the bridge to homelab services

A train dispatcher who watches the weather

A train dispatcher does not just send trains on schedule. They check the tracks, the weather, and whether the destination station is open. If there is a storm, they reroute or delay. The AgentHarness scheduler works the same way -- every 15 minutes, it checks the network before deciding what to run.

🌎

Online Window

Internet is up. Run tasks that need the web: update checks, RSS fetches, cloud LLM health probes, alert delivery.

🔌

LAN-Only Window

Internet is down but LAN works. Run tasks that only need local network: service health checks, container monitoring.

🌙

Offline Window

Everything local-only. Run heavy tasks: Docker cleanup, benchmarks, backups to USB, log rotation, security audits.

One tick of the scheduler

Every 15 minutes (via systemd), the scheduler runs exactly one tick. Here is what happens:

CODE 
def tick(self):
    write_heartbeat(self._data_dir)
    network_state = get_network_state()
    window = get_window(network_state)
    registry = load_registry(bundles_dir)

    for name, check in registry["checks"]:
        if self._cb.is_open(name):
            continue  # circuit open
        result = subprocess.run(command)
        if passed:
            self._cb.record_success(name)
        else:
            self._cb.record_failure(name)
            if restart_cmd:
                subprocess.run(restart_cmd)
PLAIN ENGLISH

One scheduler tick starts here.

First, write a heartbeat file so the watchdog knows we are alive.

Ping 8.8.8.8 to check if internet is up.

Map the network state to a window: online, offline_lan, or offline.

Load all checks and harnesses from the YAML registry.

 

Go through each health check...

If this check has failed too many times, the circuit breaker is open -- skip it.

 

Run the check command (shell command with a 30-second timeout).

If it passed...

...tell the circuit breaker it succeeded (resets failure count).

If it failed...

...tell the circuit breaker it failed (increments count).

If the check has an auto-heal command defined...

...run it immediately (e.g., "docker restart chaguli").

Deployment & Operations

How it gets installed, updated, and kept alive

Shipping cargo to an island

Deploying to a homelab is like shipping cargo to a remote island. You pack the crate on your Mac, ship it via SCP, then a local crew (the install script) unpacks it, sets up infrastructure, and verifies everything works.

1

Pack & Ship (Mac)

Code is edited on a Mac, then copied to the homelab via scp -r . rohit@192.168.29.10:/home/rohit/agentharness/

2

Install (Homelab)

Run ./install.sh which has 10 phases: discovery, apt dependencies, engine builds, model downloads, SearXNG, systemd services, benchmarks, config, log rotation, and validation.

3

Validate

python3 cli.py smoketest runs a full post-deploy check: LLM proxy, MCP gateway, Docker containers, scheduler heartbeat, and alert delivery.

The service watchdog

A cron job runs every 5 minutes and checks every critical service. If something is down, it restarts it immediately.

BASH 
# Check hermes-gateway (systemd)
if ! systemctl --user is-active \
     hermes-gateway &>/dev/null; then
    systemctl --user restart hermes-gateway
fi

# Check Docker containers
for container in chaguli mcp-gateway; do
    status=$(docker inspect "${container}" \
        --format '{{.State.Status}}')
    if [ "${status}" != "running" ]; then
        docker restart "${container}"
    fi
done
PLAIN ENGLISH

If it is not active...

...restart it immediately.

 

 

Now check the critical Docker containers one by one (Chaguli and MCP Gateway).

 

Ask Docker for the container's current state.

 

If it is not in the "running" state...

...restart the container.

 

Move on to the next container in the list.

Provider configuration and failover

Providers are configured via environment variables. Each one only activates if its API key is set. This means adding a new provider is as simple as setting one environment variable.

CODE 
# Provider init in proxy_server.py
local = LlamaCppProvider(
    endpoint="http://localhost:8081")
providers.append(local)

if os.environ.get("GROQ_API_KEY"):
    providers.append(GroqProvider())
if os.environ.get("GOOGLE_API_KEY"):
    providers.append(GoogleProvider())
if os.environ.get("CEREBRAS_API_KEY"):
    providers.append(CerebrasProvider())
PLAIN ENGLISH

 

The local LLM (Gemma 4 running on the laptop) is always added -- it needs no API key.

 

 

 

Only add Groq if its API key exists in the environment.

 

Only add Google if its API key exists.

 

Only add Cerebras if its API key exists.

...and so on for SambaNova, OpenRouter, and Ollama Cloud.

The local model is the safety net

Notice that the local LLM is always added, even without an API key. It runs on the laptop itself, so even if all cloud providers are down or rate-limited, the homelab can still answer AI requests. It is slower, but it never goes offline (unless the laptop itself has an issue, in which case the proxy auto-restarts it).

The monitoring stack

Multiple layers of monitoring ensure nothing stays broken for long.

💓

Heartbeat

Scheduler writes a timestamp every 15 minutes. If the watchdog sees a stale heartbeat (over 30 minutes), it knows the scheduler crashed.

🔄

Circuit Breaker

After 5 consecutive failures, a check is suppressed to prevent alert storms. It auto-resets when the check starts passing again.

📝

Audit Trail

Every command executed by AgentHarness is logged to exec_audit.jsonl with timestamps, exit codes, and redacted secrets.

🔐

Integrity Check

SHA-256 manifest of all core files. Detects if files were modified outside normal updates -- catches tampering or accidental edits.
